## MeowOS 知识写入机制 - 全景梳理

当前系统里有 **三个独立的持久化层**，加上一个 **审批闸门**，总共四个 moving pieces：

---

### 层 1: MeowOS 内部知识体系 (80_Knowledge/)

按**温度模型**分冷/温/热：

| 温度 | 存储位置 | 写入方式 | 谁触发 |
|------|---------|---------|--------|
| **Hot** | `habits.md` | 系统诊断 session 从 staging 提炼写入；或 `obs:` / `记住：` 快捷词直接写入 | 系统诊断 / Vincent 主动 |
| **Hot** | `84_Fitness/_state.md` + `log.md` | 训练完成后立即更新 | Vincent 报告训练 |
| **Warm** | `87_People/`, `88_Learned/`, `81_Identity/`, `82_Projects/` 等 | session 中整理写入 | Vincent 主动或 agent 处理 |
| **Cold** | `Session_Digests/` | Digest Pipeline 产出 | Vincent 说"跑 digest pipeline" |

### 层 2: ACE 机制 (CLAUDE.md 定义的自动行为)

这是凌喵的**自动感知层**，三个齿轮：

**A - 暂存 (Auto-staging)**
- 触发条件：session 中出现关于 Vincent 的新事实、身边人、风格习惯、交互 pattern、有价值知识
- 写入位置：`_staging.md`
- 主动性：**凌喵主动写，不等你说**
- 当前状态：里面有 6 条 4/8 的暂存记录

**C - 快捷词**
- 触发条件：你说 `obs:` 或 `记住：`
- 写入位置：`habits.md` 或对应 Knowledge 文件
- 主动性：**你主动触发**

**E - 系统诊断**
- 触发条件：你说"系统诊断" / "系统增强" / "优化凌喵"
- 动作：消化 staging → 提炼进 habits.md；审批 improvement-queue；消化 digest pipeline 产物
- 这是唯一的**升温通道** - cold insight 要到这里才能变成 warm/hot

### 层 3: Claude Code Auto-Memory (跨 session 记忆)

这是 Claude Code 平台自带的持久化，**独立于 MeowOS 文件体系**：

- 位置：`C:\Users\scvin\.claude\projects\d--Ai-Project-MeowOS\memory\`
- 当前有 14 条记忆（user/feedback/project/reference 四种类型）
- 写入方式：凌喵在 session 中判断有跨 session 价值的信息时写入
- 特点：**每个新 session 自动加载 MEMORY.md 索引**，所以下次开 session 凌喵就带着这些记忆

### 闸门: 改进队列 (improvement-queue.md)

这不是知识存储，是**结构性改动的审批缓冲区**：
- 什么进来：对 CLAUDE.md / agent prompts / 路由规则的改动建议
- 流程：凌喵发现时追加 → 系统诊断 session 你审批 → 才实际写入

### 你可能关心的几个点

1. **_staging 和 auto-memory 有功能重叠吗？** 有一点，但定位不同。staging 是 MeowOS 内部的观察原料，等系统诊断提炼；auto-memory 是 Claude Code 平台层的跨 session 持久化，内容偏交互偏好和元信息。

3. **87_People 怎么写入？** 按 memory 里的约束 - 那是权威存储，不能未经你指示覆写。目前文件夹是空的。
