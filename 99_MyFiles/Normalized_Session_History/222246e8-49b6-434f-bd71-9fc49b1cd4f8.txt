## 综合分析结论

### User Turn 需要剥离的内容（全部5个文件均有出现）

| 模式 | 出现频率 | 处理方式 |
|------|---------|---------|
| `<ide_opened_file>...</ide_opened_file>` | **所有5个文件** | 正则剥离标签，保留同 turn 内剩余文本 |
| `<task-notification>...</task-notification>` | 大文件（并发 agent 场景） | 正则剥离，可能跨几十行 |
| `<ide_selection>...</ide_selection>` | 已知模式，样本未命中 | 预防性剥离 |
| `<system-reminder>...</system-reminder>` | 已知模式 | 预防性剥离 |
| `<local-command-caveat>...</local-command-caveat>` | 中等文件(fb612185) | 整个 turn 通常只有这个 |
| `<command-name>` + `<command-message>` | 同上 | 剥离，slash-command 残留 |
| Skill prompt 注入（`Base directory for this skill:`） | 大文件(186bad9e)，占了~130行 | 整个 user turn 丢弃 |
| `[Request interrupted by user]` | 小文件(3c49216a) + 中文件 | 整个 turn 丢弃 |

### Assistant Turn 过滤规则

跨文件发现的核心模式：**两阶段回复** - 凌喵先发一个1-3行的dispatch/status turn，然后紧跟一个长的实质回复 turn。

规则设计：
- 在**连续 assistant turns**（中间无 user turn）的序列中，判断每个 turn 的实际内容行数（去空行）
- **≤ 3 行**且后面还跟着 assistant turn → 标记为 intermediate，**删除**
- **> 3 行**或者是序列中**最后一个** assistant turn → **保留**
- 特殊剥离：`Credit balance is too low` 这类系统错误消息

### 边界情况

1. **同一 user turn 里混合了注入和真话** - 剥离标签后保留剩余文本（如 `<ide_opened_file>` 后跟真实问题）
2. **剥离后 user turn 变空** - 整个 turn 删除
3. **连续 assistant turns 全都很短** - 保留最后一个（它通常是结论性的）
4. **`[Request interrupted by user]` 后面紧跟的是重发内容** - 中断 turn 删掉，重发 turn 保留
