## 你让我给 options 的三件事

### A. 模块间通信架构（Q21 的回应）

先纠正一个点：**Agent 之间在 Claude Code 里是可以不经主 session 直接调用的**——subagent 内部可以再 spawn subagent（嵌套调用）。但这不是免费的：

- 嵌套越深，每一层的 context 都独立计费
- 同步调用：Meeting Agent 必须等 Calendar Agent 返回才能继续，整个调用链锁住
- 结果只回到调用者，"广播"不直接

考虑到 Joyce 的 token 痛点，我推荐**混合架构**：

| 方案 | 机制 | 适用场景 | Token 代价 |
|---|---|---|---|
| **1. 共享 KB（持久状态）** | 模块直接读写 `80_Knowledge/` | 跨 session 持久信息，如客户档案、决策记录 | 低（只读 manifest） |
| **2. Scratchpad（临时握手）** | 一个 `_scratchpad/` 文件夹存阶段性交接信息 | 同一任务流内的模块协作，比如 Meeting → Calendar → Email | 中（短期共享） |
| **3. Event Queue（异步触发）** | `_events/inbox.md`，模块写"待办事件"，下次相关 agent 启动时消费 | 不需要立即响应的跨模块意图 | 极低（pull-based） |
| **4. 直接 subagent 调用** | Agent A spawn Agent B | 必须立即拿到结果的紧密耦合操作 | 高（嵌套 context） |

**我对 Joyce 用例的建议：以 1+2+3 为主，4 仅在必要时使用。**

具体到你的 Meeting 例子：
- Meeting Agent 处理完 transcript 后，把"下周需要 follow-up meeting"写进 `_events/inbox.md`（事件 2）
- 同时把 follow-up 详情写进 `80_Knowledge/meetings/[meeting_id].md`（持久 1）
- 主 session 看到 Meeting Agent 报告"已写入 1 个事件 + 1 个 KB 条目"，dispatch Calendar Agent
- Calendar Agent 启动时读 `_events/inbox.md`，处理掉自己负责的事件，更新 outlook
- Email Agent 同理

这样的好处：**主 session 始终只看到模块 ID + 极简交接元数据，不看完整内容**。每个模块在自己的 context 里加载需要的 KB 文件。

## 1. Index 不是 Manifest（你说对了，我用错词了）

你的直觉是对的。我之前混用了，正本清源一下：

## 1. 增量 indexer 是什么 + 我改主意了

**先解释「增量 indexer」是什么：**

一个 indexer 就是一段 Python 脚本，它的工作是扫描 `80_Knowledge/` 里所有的 .md 文件，把每个文件的 `路径 + 标题 + 一句话描述` 写到一个 `_index.md` 里。这样 AI 搜索时不用读所有文件，先读这个 index 就能定位。

「增量」的意思是：脚本记住「我上次跑是几点」，下次只看「比那个时间新」的文件，老的不动。

**好处：** 快。100 个文件里只有 3 个改了，只读 3 个。
**坏处：** 边界情况多——脚本崩了？删除的文件？时间戳被外部工具改了？任何一个细节漏了，index 就跟实际不一致。

---

**听完你「绝对不能错」的要求后，我改主意了。**

**新建议：v1 用「全量重建」，不做增量。**

每次 Stop hook 触发时，indexer 直接：
1. 把旧 `_index.md` 备份成 `_index.md.bak`
2. 扫描 `80_Knowledge/` 下所有 .md 文件（不管时间）
3. 读每个文件的 frontmatter + 第一段
4. 重新写出全新的 `_index.md`
5. 完成后做一次审计：「文件系统里有 N 个 .md，index 里有 N 行」，对不上就报警

**对，这样比增量慢。** 但 Joyce 的 KB 前几个月最多几百个文件，全量重建大概几秒钟，你说「可以慢、可以浪费资源，但不能错」——那就用最笨最稳的办法。

**等什么时候 KB 真的慢到影响体验了再优化成增量**，那时候已经有真实数据，可以设计得更稳。这是 v2 的事。

## 2. 写入 KB 内容的正确性（这跟 indexer 是两件事）

| 事情 | 谁负责 | 怎么保证不出错 |
|---|---|---|
| **写 KB 文件本身**（比如把 dump 里的内容拆成 47 个 .md） | Archive Agent | 见下方流程 |
| **更新 index.md** | Stop hook + indexer 脚本 | 全量重建 + 审计 |

**KB 内容写入的安全流程（v1）：**

1. Agent 读 dump 里的源文件
2. Agent 给出**预案**：「我打算创建 47 个文件，命名如下 [list]，分布在这些文件夹里 [list]」 → 用户审批（你 Q22 已选 B）
3. 审批通过 → Agent 写到一个 **staging 文件夹** `00_Dump/_processing/`，**不直接写到 KB**
4. 写完后 Agent 做自检：「我打算写 47 个，实际 staging 里有 47 个，每个 file size > 0」
5. 自检通过 → 一次性把 staging 里的文件 mv 到 `80_Knowledge/` 对应位置
6. 移动完成 → Agent 写一行到 `_write_log.md`：「2026-04-08 14:23, archive run, 47 files moved from _processing to knowledge/identity/」
7. Stop hook 触发 → indexer 全量重建 → 审计

**这样的好处：**
- 任何一步失败都可以从 staging 回滚
- 有 `_write_log.md` 做长期审计
- 用户审批挡住所有「AI 想错了」的情况
- index 重建是机械的，挡住「AI 写完忘记记录」的情况

**回答你的问题「保留当下判断吗」：** 我修正了搜索维护机制（改成全量重建），但写入维护机制是新加的（plan + staging + 自检 + log + 审批，五道门），整个组合就是为了你这条「不能漏写不能写错」的硬要求。

## 2. 关于 MemPalace / Digest Pipeline 的想法

**跟模板的关系：**

## 2. Archive Agent / Query Agent / Archive-Write Skill 的三件套

你说的这个分工我完全理解，非常干净喵。翻译成模板的设计语言：

| 组件 | 类型 | 职责 | 调用时机 |
|---|---|---|---|
| **Archive Agent** | Full Agent (Opus) | 批量处理 Dump → 拆分 → staging → 审批 → 写入 KB | Joyce 说「处理 dump」时 |
| **Query Agent** | Full Agent (Sonnet) | 搜索 KB + 读外部源 + 返回结果 | 任何模块需要查信息时 |
| **archive-write skill** | Lightweight Skill | 接收 content + metadata → 写一条 KB entry + 更新 _write_log | Query Agent 读到外部信息时调用 |

**关键的 token 节省逻辑：**
- Query Agent 读到外部文件内容 → **在自己的 context 里**调用 archive-write skill → skill 把内容写进 KB
- **下次**同样的信息被查询 → Query Agent 在 `_index.md` 里就能找到，不用再读外部源
- Archive Agent 的完整 context（拆分逻辑、staging 管理等）**只在批量处理时才加载**

**archive-write skill 大概是什么：**
- 输入：content, title, tags[], cluster, source_path
- 做的事：
  1. 生成语义化文件名
  2. 写 frontmatter（id, title, status, tags, source, last_modified）
  3. 写内容
  4. append 一行到 `_write_log.md`
  5. 返回写入路径
- **不重建 index**（那是 Stop hook 的事）

这样 Query Agent 的 context 里只多了这个 skill 的几行调用逻辑，不需要加载整个 Archive Agent 的复杂 prompt。

## inbox.md 是什么

**v1 里这个文件是预留的空壳。** 没有模块时没有事件要传递。等 Meeting/Calendar 等模块部署后才会用到。如果你觉得 v1 不需要，可以不生成这个文件。

## Hooks 的 Review

你提到 setup 要包括 hooks 的配置。我看了你的 datetime hook 设计和当前 MeowOS 的 settings.json（目前没有任何 hook 配置）。

**v1 需要的 hooks：**

| Hook | 触发时机 | 做什么 | 实现方式 |
|---|---|---|---|
| **Datetime Inject** | SessionStart + SubagentStart | 注入当前时间到 context | JScript（零依赖，Windows 内置 cscript.exe） |
| **Index Rebuild (backup)** | Stop | 调 rebuild_index.py | Python（但你说 Stop hook 不稳定，所以只是 backup） |

**关于 Datetime hook 的跨环境兼容性：**

你设计的有两个版本：
- **JScript 版**（推荐）：用 `cscript //Nologo`，Windows 内置，零依赖。Joyce 的公司电脑一定有 cscript.exe。
- **Python 版**：需要 Python 路径，Windows Store 的 Python stub 可能出问题。

**我的建议：setup wizard 默认生成 JScript 版。** 如果用户的环境不是 Windows，生成 bash/Python 版。Setup 时检测 OS 来决定。

**你还有没有别的 hook 是想设置的？** 比如：
- **UserPromptSubmit**：拦截用户输入做预处理？（你之前说 v2 再考虑）
- **PreToolUse / PostToolUse**：在特定工具调用前后做检查？

还是 v1 就只要 datetime inject + stop backup index 这两个？
