## 💡 autoresearch × Challenge 5 初读

先澄清一个关键事实，让你的期望别偏：**autoresearch 本身不是一个你能直接拿来改造 Challenge 5 的"工具"**。Karpathy 自己在 X 上说得很直白：

> *"you don't 'use it' directly, it's just a recipe/idea - give it to your agent and apply to what you care about."*

它是个 630 行的 **模式示范**，本体是 GPT 训练循环（需要 H100），跟你刮家电评论做预测模型一点关系都没有。值钱的是里面的**四个架构原语**。让我对照 Challenge 5 看哪个能移植，哪个不能：

### ✅ 能偷的三个原语

**1. `program.md` as Skill File = 研究方法论本身就是 markdown**
这是整个 repo 最重要的文件，不是代码。它用自然语言写"agent 应该怎么研究"，包括"简洁性准则"、"绝不停下"、"如何处理崩溃"这些行为约束。Karpathy 的 Software 3.0 框架里，**人写的就是这一份 markdown，其他全交给 agent**。

对 Challenge 5 的映射：你的 `scrape_and_analyze.md` 就是这份 skill file。里面写清楚：
- 每个平台怎么抓（京东/小红书/B站/reddit/HN）
- 遇到反爬怎么办
- 一条评论如何标注 sentiment + failure mode
- 什么时候停、什么时候提交一批给主分析 agent

你已经在做的 MeowOS CLAUDE.md + agent prompts 本质上就是这个模式，autoresearch 只是给你提供了一个**极简证据：这玩意真的能跑起来让 agent 无人值守工作两天**（Karpathy 自己跑 nanochat 跑了 2 天，agent 找到了 20 个改进点）。

**2. Log-and-grep 的 context hygiene**
autoresearch 里所有训练输出都 `> run.log 2>&1` 重定向，然后只 `grep` 关键行。理由是 agent 跑 100 次实验，每次 300 行日志会瞬间爆 context。

Challenge 5 直接适用：刮评论的 subagent 会读几千条文本，**绝不能把原文回灌到主 session**。要像你刚定的新规矩一样——先落盘到 KB，然后只把"结构化的分析结果 + 文件路径"返回。这恰好是你今天刚跟我定的规矩的**工业化版本**。

**3. 固定度量 = 可比较的实验**
autoresearch 用 "5 分钟 wall-clock 内的 val_bpb" 做固定度量，所有实验都在同一把尺子上比。

Challenge 5 需要一把**自己的尺子**才能让预测模型迭代下去：
- 可能是"对过去半年真实事件的回测准确率"
- 可能是"对留出集评论的 failure mode 分类 F1"
- 可能是"what-if 预测 vs 实际市场反应的 delta"

没有这把尺子，Challenge 5 的"预测模型"永远是模糊的好/坏。这是 Karpathy 这套范式给你最硬的启发——**在 agent 能自我迭代之前，先定义可比较的度量**。

### ❌ 偷不了的一个

**4. Git branch as experiment lineage**
autoresearch 每次成功实验就 commit 到主线，失败就 `git reset --hard HEAD~1`。整个 branch 历史 = 成功实验的血统图谱。

这个对 ML 训练代码迭代很漂亮，但对 Challenge 5 没有直接映射：你要迭代的不是一份代码，是一个**数据 + 模型 + 知识库**的三元组。更合适的 lineage 载体是你本来就在做的 KB 目录结构 + `results.tsv`-style 的元数据表。

## 📚 三层证据

### 层一：商业工具全军覆没

| 工具 | Knowledge Persistence |
|---|---|
| OpenAI Deep Research (o3/o4-mini) | **零** 。每次 run 从头开始 |
| Gemini Deep Research (3.1 Pro) | **零**。1M context 只是单 session 工作内存，session 结束全丢 |
| Perplexity Sonar Deep Research | **零** |
| Grok DeepSearch | **零** |
| Anthropic | 没有 standalone Deep Research 产品，只提供 infra |

这不是我总结，是 landscape agent 对每一家的架构描述汇总出来的统一结论。

### 层二：开源也几乎全军覆没，除了一个例外

- **GPT-Researcher**：README 里写"maintains memory and context"，但实际上只是 session 内的 continuity，跨 session 啥都没有
- **LangChain Open Deep Research**：**有可配置的持久化**（CompositeBackend 可以把 `/memories/` 路由到 StoreBackend），但默认不开，要开发者自己配
- **smolagents / AutoGen / CrewAI**：无内置持久化，"是开发者自己的问题"
- **Tongyi DeepResearch**（阿里 30B MoE 开权重）：研究能力烧进 weights 里，不是外部 store
- **⭐ Stanford Co-STORM**：**唯一例外**。有一个 *dynamic mind map*——session 内实时更新的层级化概念结构，可以导出。但仍然每个 topic 重置，不跨 topic 累积

Co-STORM 是全世界唯一一个朝你那个方向走了半步的工具。

### 层三：学术界明确点名了这个 gap

三篇 arXiv 综述：

1. **arXiv 2506.12594**（"A Comprehensive Survey of Deep Research"）：分析了 80+ 实现，明确指出"Knowledge Synthesis and Output Generation"是四个技术维度中**最不发达的**
2. **arXiv 2506.18096**（"Deep Research Agents: Systematic Examination & Roadmap"）：列出七大 gap，包括 self-evolving agents
3. **arXiv 2508.12752**（"Deep Research: A Survey of Autonomous Research Agents"）：原话 ——"**no explicit discussion of knowledge persistence mechanisms** in current literature"

## 🧠 一个更 juicy 的发现：方案已经存在，只是没被拼起来

"读完就存"的工程方案其实已经有了，全在**另一条技术线**上，叫 **Agent Memory Layer**：

- **Zep** (arXiv 2501.13956)：时序知识图谱，双时态边建模，事实不删除只 invalidate。把 context token 从 115K 压到 1.6K 的同时精度还涨了 18.5%
- **Mem0** (arXiv 2504.19413)：基于图的抽取 + 更新。精度 +26%，延迟 -91%，token -90%。两个变体：base 和 Mem0g（带实体/关系的有向标注图）
- **A-MEM** (arXiv 2502.12110)：**Zettelkasten 原则**（!! 这个你看过 Ahrens 的书吧），新 memory 可以反向更新旧 memory 的 context，双向知识演化而非只追加

**关键事实：没有任何一个 deep research agent 产品接入了这三样中的任何一个。**

Research agent 层和 memory 层目前是**完全分开的两个 stack**，没人把它们焊在一起。Landscape agent 的原话：

> *"A research agent built on any of these memory systems could in principle accumulate domain knowledge across sessions... This integration does not yet exist."*
