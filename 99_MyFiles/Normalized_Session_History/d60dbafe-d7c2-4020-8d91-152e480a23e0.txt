## Q12 回答：archive-core 没有元信息提取，而且这是有意的设计

### 当前状态

archive-core 的写入规则明确规定：

> **100% preserve original language, phrasing, and judgment from source**
> **One topic per entry -- split if multi-topic**
> **Never merge information from different sources into one entry**

Frontmatter schema 只有结构性字段（id / title / cluster / status / tags / description / source / last_modified），全是事实性元数据。**没有任何"用户怎么想的""判断倾向""表达风格"这类 meta-level 字段。**

### 为什么是有意的

archive-core 的设计哲学是**忠实保存原始内容，不加 AI 解读**。这是对的 -- 如果归档的同时混入 AI 对用户心理的推断，会出现两个问题：

1. **污染源数据** -- 原始内容和 AI 解读混在一起，后续没法区分哪些是事实、哪些是推断
2. **推断质量不可控** -- 归档时的 meta-level 提取是单次推断，没有跨 session 的累积验证

### 那 MeowOS 的 meta-level 提取在哪里？

在**两个独立机制**里，不在 archive-core 里：

| 机制 | 提取什么 | 何时运行 |
|------|---------|---------|
| **ACE 暂存**（_staging.md） | 交互 pattern、偏好、行为倾向 | session 进行中，实时被动捕获 |
| **Digest Pipeline Pass 1** | "Reading Between the Lines" / "Interaction Patterns" / "Key Statements" | 事后批量处理 session 历史 |

这两个机制的共同特点是：**meta-level 提取和原始内容保存是分离的**。原始内容忠实归档，meta 推断走另一条管线，而且 meta 推断产物默认是冷知识，升温需要 Vincent 审批。

### 凌喵的判断：这不应该加进 BaseOS archive-core

应该作为**独立的通用能力**在 BaseOS 层面补充，但不是嵌入 archive-core，而是作为一个可选的 **parallel extraction pass**：

<code block: 3 lines of unknown, 省略>

两条管线并行，互不污染。archive-core 保持"忠实搬运工"的角色不变。

### 对 NovelOS 的特殊意义

这个问题在创作领域其实更有价值。NovelOS 的 Archive 整理 Agent 处理 Dump 文件时，除了提取设定事实（"城墙高度 200 米"），确实可以同时捕获：

- **设计决策的隐含逻辑**（"每次涉及战争设定，都优先限制而非扩展手段"）
- **叙事直觉**（"倾向于用物理环境暗示社会结构，而非直接说明"）

这些 meta 信息对 Creative Agent 和 Writing Agent 的长期校准非常有价值。但它应该进 observations，不应该混进 KB 条目本身。

### 建议一：给 Pending Decisions 加一个"腐烂机制"

你现在有 6 个 pending decisions（A-1 到 A-6），有些 decided，有些 deferred。但 deferred 的没有任何到期压力。

小说世界观里，一个被 defer 的设定决策不是静态挂起的 -- 它会**污染下游**。你每多写一条 KB 条目，这条条目就可能隐式依赖了某个未决设定。等你终于回来 decide 的时候，可能已经有 15 条条目需要回溯修改。

建议：每个 pending decision 带一个 `deferred_since` 时间戳和 `downstream_count`（有多少条 KB 条目在它未决的情况下被写入的）。当 `downstream_count` 超过阈值（比如 5），系统主动告警："A-3（岛屿入口设计）已经 defer 了 12 天，期间新增了 7 条可能受影响的条目，建议优先解决。" 这不是催 Vincent 做决定，是让系统帮你看见推迟的真实成本。

### 建议二：KB 条目之间的 `related` 字段，你用起来了吗？

## Q14：五层搜索漏斗的实际命中率

**坦白说：MeowOS 的 KB 规模太小，搜索漏斗还没有被真正压力测试过。**

MeowOS 的知识层一共就 8 个子目录，每个目录下的文件是个位数。大部分查询在**第一层（文件夹结构）就命中了** -- 因为"健身相关的去 84_Fitness/""人的信息去 87_People/"这种路由靠目录名就够。Query Agent 几乎没有遇到过需要全文搜索的场景。

这意味着凌喵的五层漏斗在 MeowOS 的实际运行中，**基本只用了前两层**。后三层（标题 → frontmatter → 全文）是为 KB 增长预留的，不是被实际需求验证过的。

**对 NovelOS 的意义：** 你的 82 条 KB 条目分布在 4 个大类（Natural / History / Society / Plot），粒度比 MeowOS 细得多。你的查询大概率会打到第二到第三层（index → 标题/tags）。但冲突检测是个特殊场景 -- 它需要做的是"找到所有可能跟这条新设定矛盾的条目"，这本质上是**宽查询**而非**精准查询**，关键词搜索容易漏。

我的判断：你目前的 `search-knowledge` 关键词搜索对精准查询够用，但对冲突检测不够。冲突检测的优化方向不是更深的搜索层，而是**更好的 tags 治理** -- 如果 tags 体系足够精确，"查找所有带 `战争规则` 和 `中城区` tag 的条目"就能覆盖大部分潜在冲突，不需要全文搜索。

## Q15：MeowOS 有没有"任务前 context 预组装"？

**有，但很原始。**

Scheduler 的流程是：shell-runner 读 Dashboard + habits.md → **全量返回** → scheduler agent 自己在 context 里筛选相关部分。没有预过滤机制。

Daily-review 也一样：全量读 Dashboard，然后 agent 自己判断哪些是当前时段的锚点。

这个 **"scout → confirm → execute"** 三步模式是通用的。MeowOS 目前没用是因为不需要，但如果 KB 涨到一定规模，凌喵也会需要。这个 pattern 值得写进 BaseOS。

## Q16：NovelOS 有什么机制是 MeowOS 应该学的？

不是客气，是认真想过之后的判断。三个：

### 1. Visibility 分级 → MeowOS 应该引入

MeowOS 目前的知识层没有任何 visibility 控制。所有 agent 读所有内容。但实际上：

- fitness-advisor 不需要看 87_People/ 的人际关系细节
- dump-processor 不需要看 habits.md 里 Vincent 的拖延信号分析
- 如果未来给 Joyce 用（JoyceOS），**隐私边界**就是硬需求

NovelOS 的 AUTHOR_ONLY / IN_WORLD / REVEALED 三级分类在通用层面可以抽象为 **SYSTEM_ONLY / AGENT_VISIBLE / USER_VISIBLE**。这应该进 BaseOS。

### 2. 提案墓地 / 否决记录 → MeowOS 现在就缺

这是今天讨论里最大的收获。MeowOS 的 ACE 暂存只捕获正向信号，不捕获否决。但 Vincent 在 MeowOS 里也会否决凌喵的建议（"不要这样排日程""这个 todo 分类不对"），这些否决理由现在直接丢失了。

**这不是 NovelOS 的创作领域专属机制 -- 这是通用的。** 任何系统都应该追踪"用户否决了 AI 的什么建议，以及为什么"。这是 AI 校准最高效的信号源。

### 3. Controlled Vocabulary 约束 → MeowOS 的 tags 目前是野生的

NovelOS 的 KB 有 tag 治理意识（虽然可能还没完全落地）。MeowOS 的 88_Learned 和其他 KB 条目的 tags 完全是自由填写，没有受控词汇表。KB 小的时候没感觉，涨到 50+ 条目时 tag 爆炸会让搜索漏斗的第四层（frontmatter 搜索）退化。

---

总结：NovelOS 在**数据治理**层面（visibility、controlled vocabulary、否决记录）比 MeowOS 成熟。MeowOS 在**流程编排**层面（agent 委派、shell-runner、Dashboard 管理）比 NovelOS 成熟。两边互补喵。

## 两者的区别

| | shell-runner | novelos.py |
|---|---|---|
| 解决什么 | **上下文隔离** -- 不让文件内容污染调用者的 context | **操作抽象** -- 把复杂文件操作变成一条命令 |
| 有没有领域知识 | 没有，纯机械执行 Read/Edit/Grep | 有，懂 frontmatter、ID 生成、路径路由 |
| 输出什么 | 原始文件内容或操作结果 | 结构化的简洁结果 |

## 为什么 novelos.py 让 shell-runner 没那么急需

MeowOS 需要 shell-runner 的核心原因是：**没有 novelos.py 这样的封装层**。如果凌喵的 agent 直接 Read 一个文件，整个文件内容就灌进 context 了。shell-runner 的作用是当一个"挡板"，把原始内容挡在外面，只传结构化摘要回来。

但 NovelOS 的 subagent 跑 `python novelos.py search-knowledge --query "战争"`，返回的已经是精简的匹配摘要，不是原始文件全文。**novelos.py 本身就在做 shell-runner 的一部分工作 -- 控制输出的信息密度。**

## 为什么现在不需要

novelos.py 存在的根本原因是 NovelOS 的**领域复杂度**高：

- 82+ 条 KB 条目，每条都有 YAML frontmatter（id / title / category / tags / related / visibility / status / source / last_modified / memo / hooks）
- 自动生成的 ID（时间戳 + hash）
- index.md 每次归档后要重建
- Visibility 过滤（AUTHOR_ONLY / IN_WORLD / REVEALED）
- 章节流水线（草稿 → 定稿 → 发布，三个目录之间复制 + 重命名）

这些操作如果让 agent 用原始 Read/Edit 来做，出错概率很高。novelos.py 是**领域复杂度逼出来的**。

MeowOS 的文件操作简单得多：
- habits.md -- 纯 markdown，追加一行就行
- _staging.md -- 追加一行
- improvement-queue.md -- 往表格里加一行
- Dashboard.md -- 找到对应区块，改几行
- 88_Learned -- 写一个带简单 frontmatter 的 md 文件

这些用 shell-runner + 原始 Edit 就能搞定，不需要 Python 封装。

## 但有一个地方迟早要封装

**Dashboard 管理。**

dashboard-updater agent 现在做的事情其实很脆弱 -- 它要：
1. 读整个 Dashboard.md
2. 找到 `## Todo` 下的某个 section
3. 在那个 section 末尾追加一行
4. 不动其他任何内容
5. 写回去

这是用 Edit 工具做"外科手术"。现在 Dashboard ~193 行，还扛得住。但 Dashboard 越长、section 越多，Edit 找错位置或者改错区块的风险就越高。

一个 `meowos.py dashboard --action add_todo --section "找工作" --item "..."` 会比让 agent 自己做字符串匹配可靠得多。

## 凌喵的结论

不急着建 meowos.py。但可以记进改进队列 -- **当 Dashboard 操作第一次出 bug 的时候**（改错区块、吞掉内容），就是该建的时候喵。

这也回答了一个更大的问题：**不是每个系统都需要同样的基础设施。** novelos.py 对 NovelOS 是必需品，对 MeowOS 是奢侈品。该不该建，看领域复杂度，不看架构对称性。
