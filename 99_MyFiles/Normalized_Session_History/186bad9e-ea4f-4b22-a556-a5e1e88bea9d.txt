## 方案 A：增强版单 Agent + 结构化知识文件

在现有 `fitness-advisor.md` 基础上大幅升级 prompt，新增一组知识文件（健身房档案、伤痛约束、备餐注册表、营养日志等），agent 每次被唤起时读取这些文件做推荐。

- **缺点**：单个 agent prompt 会很长很重；营养和训练的分析逻辑混在一起；随着数据积累，每次调用要读的文件越来越多

## 方案 B：模块化知识系统 + 职能分工（推荐）

把健身体系拆成**知识层**和**agent 层**：

**知识层**（`84_Fitness/` 下新增文件）：
- `gyms/` — 三个健身场所的器材档案，你说"今天去阳台"agent 就知道能用什么
- `constraints.md` — 伤痛约束 + 动作偏好（反握不行、不频繁换片等），你随时可以追加
- `programs/` — 四条目标线的训练模板（周期化编排框架），agent 在此基础上做每日适配
- `nutrition/` — 备餐注册表（配方×份数）+ 每日饮食日志 + 宏量素目标
- `log.md` — 升级为结构化格式，支持后续分析

**Agent 层**（两个 agent）：
- `fitness-coach` — 日常训练推荐 + 训练记录。读取健身房档案 + 约束 + 程序模板 + 状态，输出当日方案

## 周容量 Volume Cycling 解释

现在spec里每周目标是8个movement pattern**同时**打满容量（比如H-Push 12-18组、H-Pull 12-14组……全部同时进行）。

Volume cycling的意思是：**同一时间只focus 2-3个pattern拿满容量，其余降到维持量（5-10组）。** 几周后轮换focus。

为什么这样更好：
- 你每周训练时间有限，8个pattern全打满 = 每个都只能勉强够，或者session时间爆炸
- 维持量（5-10组）已经能保住85-90%的效果，不会退步
- Focus pattern拿到充足刺激，真正能进步
- 对关节（特别是TFCC）的总负荷更可控

**举例：** Hypertrophy阶段前4周focus H-Push + Hinge + V-Pull，其余维持；后4周换成 V-Push + H-Pull + Knee Dom，其余维持。

这其实很符合你的生活模式——你本来就不是每周都能均匀练到所有东西的人。Volume cycling把这个现实变成一个**有意识的设计**，而不是随机的遗漏。

**你觉得这个逻辑make sense吗？要加入spec吗？**

这个观察非常好喵。其实这恰好让volume cycling更有道理：

**核心矛盾：** 你不是每周稳定练5次的人。可能这周练4次，下周只练2次。

**Volume cycling + 你的模式的解法：**

每次session设计成一个**自足单元**：focus patterns拿到足够刺激（高容量），maintenance patterns做几组保底。这样——

- **去了3-4次**：focus patterns进步，maintenance patterns不退步，完美
- **只去了1-2次**：focus patterns至少有1-2次高质量刺激（总比8个pattern都浅尝辄止好），maintenance也沾到了
- **去了5次**：多出来的session可以给maintenance patterns加点量，或者做mobility/TFCC prehab

**关于"练狠了后面不去"的平衡：**

凌喵的思路是给每个session设一个**时间上限**（比如60-75min含热身），而不是无限堆容量。在固定时间内，focus patterns优先分配组数，剩余时间给maintenance。这样：
- 每次去都练得充实、有成就感
- 但不会因为"今天要补昨天的量"搞成2小时马拉松式训练
- session结束时的感觉应该是"练到了"而不是"被榨干了"

**写入spec的原则：**
1. Session设计按"focus + maintenance"结构，不按"每周总量平摊"
2. 单次session时间上限60-75min
3. 一周只练1-2次时，agent不焦虑不补量，focus pattern依然拿该拿的刺激
4. 跳过的天数不产生"欠债"——不存在"这周少练了所以下周加倍"

这样volume cycling就变成了一个**容错机制**而不是刚性计划。make sense喵？
