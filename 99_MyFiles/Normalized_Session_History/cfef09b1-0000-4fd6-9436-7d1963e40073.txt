<code block: 7 lines of unknown, 省略>

就这。Tool call 的 JSON、tool result、思考链、系统提醒——**全部不要**。

> 注意：这意味着 tool 调用结果（比如我刚才 grep 出来的内容、文件内容）也都丢了。它存的是"对话"，不是"工作记录"。

### 第二步：chunking（切块）

`convo_miner.py` 把上面那种纯净文本**按"问答对"切块**——一个 user turn + 紧跟的 assistant reply 算一个 chunk，大概几百到一两千 token。

每个 chunk 同时被打上标签：
- 属于哪个 wing（项目/人）
- 属于哪个 room（话题，靠 `entity_detector.py` 自动识别）
- 属于哪个 hall（事实/决策/偏好/事件...，靠 `general_extractor.py` 分类）

### 第三步：存进 ChromaDB

每个 chunk 同时存两份：
1. **原文**（drawer）—— 完整问答对的纯文本
2. **embedding 向量** —— 用本地 embedding 模型把这段文本编码成 384/768 维向量

检索时，你的新问题也被编码成向量，去库里找**余弦相似度最高**的几个 chunk，把对应的原文拉出来。

---

## 回到你的问题："它是怎么做到读原文的？"

不是"读"，是**"提前消化好，按需取片段"**。

| 阶段 | 数据形态 | 体积 |
|------|---------|------|
| 你硬盘上的 jsonl | 90% 是元数据噪音 | 1.6 MB |
| normalize 之后 | 纯对话文本 | ~150 KB |
| 切块进 ChromaDB | 文本 + 向量 | ~300 KB（含索引） |
| 单次检索注入 context | top-N 个相关问答对 | 1,500–4,000 token |

它**永远不读完整 jsonl 给 LLM**。LLM 看到的永远只是检索出来的几个最相关片段。

## 废话 session / 进行中 session 怎么办

**核心解法**：让 digest agent 自己判断并**明确标注状态**，而不是盲跑生成。每份 digest 头部加一行 `status:` 字段：

<code block: 4 lines of yaml, 省略>

然后：

| Status | Pass 1 产出 | Pass 2 处理 |
|---|---|---|
| substantive | 完整 digest | 正常参与思想脉络分析 |
| operational | 只写一行"做了什么"，不深挖 | 跳过，不进入合成 |
| in_progress | 完整 digest + **open_threads 字段** 列出未完结问题 | 专门收集 open_threads，单独一节"正在进行中的思考" |
| fragment | 一行略过 | 跳过 |

**进行中的 session 最关键的点**：Pass 2 不应把它当"结论"，而应把它当"某条长期思考的当前切片"。所以合成时会尝试**把同一条线头跨 session 串起来**——比如你在 session A 埋了一个想法，session C 又变形出现，Pass 2 要能识别出"这是同一条线在不同阶段的投影"。

## 金句为什么出现 —— 你的观察是准的

先说结论：**你感觉有些"金句"没金句味儿，不是你的问题，是 prompt 的问题**。

模型在执行时会把这一条拆成**至少五种**不同的"引用理由"，而且它们会**混在一起**交差：

| 模型眼里的"值得记录" | 实际是什么 |
|---|---|
| ① **Voice 类** | 能体现你说话方式的句子——不必有智慧，只要有"你的味儿" |
| ② **Views 类** | 明确的立场、观点、态度——"不要让他们被 Claude 惯坏了" 这种 |
| ③ **Sharp moments 类** | 修辞上有张力、有意外的句子 |

| ⑤ **Diagnostic 类** | 陈述事实的句子（"Python 在 Windows 上被劫持了"）——模型把"重要信息"和"值得引用"搞混了 |

你看的时候觉得"这算什么金句"的那些，几乎全是 ④ 和 ⑤。它们被选进来**不是因为那句话本身值得保留**，而是因为**模型在用引用帮自己搭结构**。

## 更本质的问题

人类说的"金句"带着一种**无法被转述**的质感——你改一个字就不对了，你用自己的话说就掉味了。这种"不可转述性"其实就是**共鸣**。

模型没有共鸣。它只能用几个可观察的代理指标去近似：
- **信息密度高**
- **用词独特**
- **语气强烈 / 明确立场**
- **自带修辞结构**（对仗、否定式、三段论）

所以模型倾向于选那些"**看起来像金句的句子**"，不是"真正金句的句子"。特别是 Sonnet 这种讨好型的中庸模型，它会在每份 digest 里机械地凑够 5-8 条，宁多勿缺，稀释掉了真正有味道的那几句。

## 5. Watershed 是什么 + 怎么标

先解释概念。"Watershed" 字面意思是**分水岭**——在这套 pipeline 里我用它指**值得做二阶分析的 session**。不是所有 substantive session 都值得 Pass 2，只有那些**有深层内容、值得看字面之下**的才值得。

举个例子，刚才我们做的 `2b525bc7`（时间 hook + 给 Joyce 写文档）——Pass 1 digest 已经很丰富了，Pass 2 真的挖出了新东西（"情感工程"、"制品工厂"那些）。这种叫 watershed。

但如果一个 session 是纯技术调试，Pass 1 digest 就够了，Pass 2 只会挤出一堆注水的"哲学解读"。这种就不是 watershed。

**谁来判断？** 我原本想的是**你自己读完 Pass 1 后手动标**——你读了 digest 最清楚哪些有深度。

**怎么标？** 三个选项：

| 方案 | 操作 |
|---|---|
| A. manifest 里加一列 `watershed`，你在里面打 ✓ | 所有状态集中在 manifest |
| B. 独立文件 `watershed.txt`，一行一个 uuid | 更简单，分散 |
| C. 跟凌喵直接说 "跑 Pass 2，针对 2b525bc7 和 d63100fa" | 完全 ad-hoc，不需要文件 |

我倾向 **C**——既然流程是"你跟凌喵对话"，不需要另建文件系统。manifest 只记 Pass 0/1 状态，Pass 2 按需触发。

### 先说分层

我现在的记忆系统有四层（从冷到热）：

| 层 | 位置 | 加载时机 | 影响 |

**关键分水岭在 Warm 和 Hot 之间**。

### Pipeline 产物的默认归宿

v1 产物**只落在 Cold insight 层**。也就是：

<code block: 2 lines of unknown, 省略>

### 为什么不自动写 Warm / Hot？

因为**自动写会污染你**。

- AI 的归纳可能有微妙的偏差，写进去后会**反过来影响凌喵怎么理解你**，形成回音室
- 一旦写入，你未来每次说"Human Becoming" 时，凌喵都会带着已经写入的那个版本的理解来回应——即使那个版本是错的

**时机一：系统诊断 session**

**时机二：你的手动触发**

## 1. Digest Pipeline = Dream Mode 的 Gather + Consolidate

你的 dream-mode 研究里的四阶段对照：

| Dream Mode 阶段 | 我们建了什么 | 还缺什么 |

**结论**：digest pipeline 把 Gather 和部分 Consolidate 覆盖了，但 **Orient 和 Prune 是完全空白的**。这两个应该加进系统诊断的流程里。

## 3. "Don't exhaustively read, grep for what you suspect matters"

这条原则和 digest pipeline 的关系很妙——我们的 pipeline 是**全量读一次然后再也不读了**（Lossy by Design），而 Dream Mode 的 Gather 是**每次只 grep 你怀疑有价值的东西**。

两者不矛盾，是**不同频率的两种采集**：

- **Pipeline**（低频，每隔一段时间）：全量扫一遍新增 session，做结构化 digest，存冷归档
- **Dream Mode 式 grep**（高频，每次系统诊断）：针对 `_staging.md` 暂存里出现的关键词，去 grep 最近几天的 normalized txt，补捞可能遗漏的观察

## 缺失的

**没有"回头验证"机制**。如果 Pass 1 对你说过的话做了错误推断，目前没有任何自动化的东西会拿 digest 去和原始 transcript 对比验证。唯一的验证者是你。

**没有"标记为错误"的传播机制**。你告诉凌喵"这个 digest 不对"，但那份 `pass1/<uuid>.md` 文件本身不会被修改或标记。如果未来某次 Pass 2 又读到了它，同样的错误可能再次被引用。

**没有 digest vs 现有 memory 的对比**。Pass 2 只看 digest 之间的矛盾，不看"digest 和 MEMORY.md 里已有的条目是否矛盾"。

---

## 最简单的补丁（不需要现在做，放这里你知道就行）

1. **在 manifest 加一列 `verified`**——你读完一个 digest 觉得 OK 就标 `✓`，觉得有问题就标 `⚠` + 写 notes。以后 Pass 2 的 prompt 可以加一条"如果 manifest 标了 ⚠ 的 session，降低该 digest 的权重"。

2. **系统诊断的 Consolidate 阶段加一步**：把候选写入条目和现有 MEMORY.md 条目做显式对比，发现冲突时列出来让你裁决，而不是静默覆盖。

3. **最极端的**：对某个你觉得"写得很离谱"的 digest，派一个 subagent 拿着 digest 回去读原始 transcript 验证。成本高，但精准。按需触发，不自动。

---

**总结一句话**：目前的纠错本质上是**靠你当最后一道闸门**。pipeline 能做的是把矛盾尽量在 Pass 2 里浮出来，但判断对错的能力只有你有。这不是缺陷——这是 Lossy by Design 的一部分：**宁可漏掉正确的，也不自动写入错误的**。
