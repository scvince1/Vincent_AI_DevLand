## 1. Summary

The MacBook harness is structurally close to working: `CLAUDE.md` and all three dispatchable subagent prompts are English, frontmatter is valid, the symlinks in `.claude/agents/` resolve correctly, and `settings.json` is valid JSON. Path references from `CLAUDE.md` and the agent prompts are almost entirely intact (only one dangling reference found). The memory index is consistent with the memory directory. The main risks are not wiring bugs but (a) historical Chinese-language carryover inside `80_Knowledge/` files that are named / referenced by the agent system (e.g. `pending_candidates.md`, `improvement-queue.md`, `_staging.md`, `habits.md`, `_rules.md`) and (b) one routing/trigger mismatch where the stale `pending_candidates.md` expects a Chinese Phase 2 trigger while `system-diagnostics.md` specifies an English one. Subagent discovery-at-session-start is a known constraint — the symlinks were placed mid-session, so they will not dispatch until the next session.

## 5. Rule conflicts or ambiguities

- **`obs:` shortcut vs. shell-runner rule.**
  - `CLAUDE.md` line 39: "All file reads and writes … are delegated to the `shell-runner` subagent."
  - `CLAUDE.md` line 58: "C · Shortcut — `obs:` / `remember:` — Write immediately to `habits.md` or the matching knowledge file."
  - `01_Routing.md` line 11: "`remember this` / `obs:` → main session handles directly (append to `83_Observations/habits.md`)."
  These three read, together, as "main session writes directly" for `obs:`. That contradicts the shell-runner rule unless the `obs:` case is an implicit exception. Intent seems to be "it's a tiny append; do it inline" but the rule is not spelled out. Recommend making the exception explicit.
- **CLAUDE.md priority ordering vs. routing exception list.**
  - `CLAUDE.md` line 32: priority is "Forbidden actions > ACED > shell-runner > workflow."
  - `CLAUDE.md` lines 41–43 list two exceptions to the shell-runner rule (explicit Vincent ask; coding / complex-system session).
  - `01_Routing.md` lines 35–37 repeat the same two exceptions.
  - Consistent, but the priority ladder does not mention where the exception list sits. If "Forbidden actions" ever bans something that one of the exceptions would otherwise permit, there is no tie-break rule written down.
- **`knowledge-agent` "execution principle 3" vs. its own tool list.**
  - `knowledge-agent.md` line 59: "All file operations go through `shell-runner`. Do not call Read / Edit directly inside this agent."
  - But the agent's frontmatter grants it `Read, Write, Edit, Glob, Grep` directly (line 4).
  - Claude Code subagents cannot call other subagents, so "go through shell-runner" here is actually infeasible — the rule is internally inconsistent. Probably carryover from a prior architecture where knowledge-agent delegated further. Flag for rewrite.
- **`system-diagnostics` Phase 1 read list vs. shell-runner rule.** `system-diagnostics.md` line 30 says "Read all relevant knowledge files (via `shell-runner`)" — same infeasibility as above: Claude Code subagents cannot dispatch other subagents. The diagnostics agent must read with its own tools. Document or rewrite.
- **Routing table orchestration rule vs. actual subagent capability.** `01_Routing.md` line 24: "When an agent emits a `cross_agent_required` flag, the main session dispatches to the listed `target` agents **sequentially**." None of the three existing subagent prompts actually define or emit a `cross_agent_required` field. The principle is fine but currently aspirational.
- **"Writes are forbidden in Phase 1" vs. "the candidate file is the only write allowed".** `system-diagnostics.md` lines 21–23 + 52–53. Internally consistent (candidate file is called out explicitly as approval material, not a system file), just tight and worth flagging so Vincent notices the exception.

## 6. End-to-end workflow trace results

### Workflow A — Vincent says `obs: I prefer X`
**Verdict: partial.**
- Trigger matches `01_Routing.md` line 11 and `CLAUDE.md` line 58.
- Destination file `83_Observations/habits.md` exists.
- However, routing says "main session handles directly," while CLAUDE.md's shell-runner rule says "no direct writes" and the exception list (explicit ask / coding session) does not obviously cover `obs:`. In a strict reading, the main session does not know whether to write inline or dispatch to `shell-runner`. Both behaviors produce the same end file, so the user gets their observation logged — but the agent is in a weakly-specified branch. Not dead, but ambiguous. Recommend: make the `obs:` inline-write exception explicit in CLAUDE.md.

### Workflow B — Vincent says `run a system diagnostic`
**Verdict: partial → probable dead-on-arrival on the first attempt.**
- Trigger matches `01_Routing.md` line 9 and `system-diagnostics.md` description.
- The agent exists and frontmatter is valid, so it can be dispatched (after the symlink-discovery session boundary — this session was flagged as not-yet-dispatchable).
- Phase 1 read list (lines 30–37) references `_staging.md`, `habits.md`, `_rules.md`, `_state.md`, `improvement-queue.md`, `CLAUDE.md`, `01_Routing.md`, `90_Agents/*.md` — all exist.
- Phase 1 is told to write to `85_System/harness_engineering/audit/pending_candidates.md` — that file already exists with stale content from a different agent roster. The agent will overwrite it, which is probably fine, but a Phase 2 invoked against a leftover pre-write copy (the current state) would execute obsolete candidates written for a different system.
- Instruction line 30 "read … via `shell-runner`" is infeasible from inside a subagent (see §5). The agent will either fall back to its own `Read` tool (likely) or hallucinate a shell-runner call (bad).
- Phase 2 trigger word in the existing `pending_candidates.md` (`执行系统诊断 Phase 2`) does not match the agent prompt's English trigger. If Vincent copies the stale file's instruction, nothing fires.
